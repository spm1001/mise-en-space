The remote mode architecture (--remote flag, StreamableHTTP, inline content) was designed around a principle: don't touch the fetchers. The 10+ type-specific fetch functions in tools/fetch/ are battle-tested and complex — adapting them for remote would have been high-risk refactoring. Instead, the server.py layer intercepts after the fetch completes: deposits go to a temp dir, content is read back and included inline in the response, temp dir is cleaned up. This "post-hoc read-back" pattern means all fetchers work unchanged in both modes. The same principle applies to search — SearchResult.to_dict() already had an inline mode (path=None), so remote just clears the path after the deposit. When extending this pattern (e.g., for new content types), don't add remote-awareness to the fetchers themselves — keep the interception at server.py.

The operation gating for remote mode uses a whitelist (_REMOTE_ALLOWED_OPS), not a blacklist. Error messages list only allowed ops, never the restricted ones — this prevents leaking the existence of operations that aren't exposed. The tool description also adapts per mode via a conditional `description=` parameter on @mcp.tool(), which requires _REMOTE_MODE to be set at module load time (before decorators run). This early-evaluation pattern is intentional and documented in CLAUDE.md's Gotchas — don't refactor it without understanding the timing constraint.
