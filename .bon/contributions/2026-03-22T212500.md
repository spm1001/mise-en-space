Image embedding in Google Docs faces a hard constraint: the Docs API `insertInlineImage` requires a publicly accessible HTTPS URL — there's no "insert from Drive file ID" equivalent (Slides API has one, Docs doesn't). The current approach uploads temp images to Drive, shares publicly for seconds, inserts, then revokes. This works for personal accounts but will 403 on enterprise Workspace accounts with DLP (Data Loss Prevention) policies that block `type=anyone` sharing. The graceful degradation path (skip image, report in `cues.image_errors`) is tested in unit tests but the entire happy path is mocked — the batchUpdate request format, the `uc?export=view` URI, and the permission lifecycle have never hit real APIs. If enterprise support becomes necessary, GCS signed URLs is the clean alternative (time-limited public URL without Drive sharing semantics).

The multipart upload boundary in `http_client.py` was changed from a static string to UUID-based (`mise_{uuid4}`) because binary file uploads (via the new `file_path` parameter) could theoretically contain the boundary string in their content. The old boundary was fine for text uploads where collision was astronomically unlikely.
