## For Claudes to come

**Batch ops + pagination are the infrastructure pair that makes inbox triage viable as an MCP workflow.** Neither alone is sufficient — batch without pagination means you can act fast on a partial picture; pagination without batch means you can see everything but act painfully slowly. The combination (this session: 80 threads fetched, 18 archived in 3 tool calls) is what crosses the threshold from "demo" to "useful." The sequential-per-thread implementation is the right call — Gmail's `messages.batchModify` works at the message level not thread level, so using it would require resolving thread→message IDs first (one extra API call per thread), netting zero gain for typical triage volumes.
