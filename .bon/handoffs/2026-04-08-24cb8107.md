### Risks
- **Pageless field untested against live API.** `documentFormat.documentMode` was found in reference HTML, never hit real endpoints. If the field name or casing is wrong, `_set_pageless` will throw, but graceful degradation means the doc is still created — pageless just silently doesn't apply. Check `cues.page_setup_error` if it happens.
- **file_path on overwrite reads as UTF-8 always.** Binary plain file overwrite via `file_path` (e.g. replacing a PNG) would fail with a UnicodeDecodeError. The `source` path reads bytes for plain files, so there's an asymmetry. The use case is markdown files — binary overwrite via file_path is unlikely but worth knowing about.
- **Tab filtering doesn't reduce chart rendering.** `tabs` filters the value batchGet but charts are per-spreadsheet. A 14-tab sheet with charts on every tab still renders all chart PNGs even when you only want one data tab.

## For Claudes to come

The Google Docs API v1 `DocumentStyle` has a `documentFormat` field containing `documentMode` — an enum with values `PAGES`, `PAGELESS`, and `DOCUMENT_MODE_UNSPECIFIED`. This is the correct way to set pageless mode programmatically. The batchUpdate request uses `updateDocumentStyle` with `fields: "documentFormat"` and `documentStyle.documentFormat.documentMode: "PAGELESS"`. Do NOT use `useCustomHeaderFooterMargins` — that controls header/footer margin behaviour only, not the page layout mode. The field was found by fetching the Docs API REST reference page and searching for "pageless" in the HTML. Since Google's API reference pages are JS-rendered, the HTML may look different depending on how you fetch it — `curl -sL` with redirect-following worked here. The `developerknowledge.googleapis.com` API would be cleaner but requires a valid API key scoped to that service.
