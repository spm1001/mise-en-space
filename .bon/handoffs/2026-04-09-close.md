## For Claudes to come

**Desktop has three separate systems for extending Claude, and they don't share plumbing.**

1. **MCPB extensions** (`.mcpb` / `Claude Extensions/`) → `LocalMcpServerManager` launches the server on the Mac. Tools bridge into Cowork. Skills in the bundle are ignored.
2. **Uploaded plugins** (`Upload Plugin` / `rpm/`) → `RemotePluginManager` loads skills into Cowork. MCP servers in `.mcp.json` are read and displayed in the UI but never launched.
3. **`claude_desktop_config.json` → `mcpServers`** → launched by Desktop, bridged as "connectors" into all modes. The simplest path for personal use.

The Cowork VM is a sandboxed Linux ARM64 VM (gvisor networking, `allowedDomains=1`, MITM proxy, ephemeral). It cannot reach the Mac's local network (`172.16.10.1` is unreachable). All MCP servers that work in Cowork run on the Mac side and are bridged in by Desktop.

The MCP protocol handshake works correctly for both MCPB and `claude_desktop_config.json` paths — Desktop sends `tools/list`, `resources/list`, `prompts/list` and receives correct responses. Resources are listed but not forwarded into the VM (tools are forwarded, resources aren't). This is visible in `~/Library/Logs/Claude/mcp-server-*.log`.

**The plugin MCP gap might not be architectural.** The plugin spec (`cowork-plugin-management/skills/create-cowork-plugin/SKILL.md`) describes `.mcp.json` with `${CLAUDE_PLUGIN_ROOT}` for stdio servers. The UI shows the connector. Permissions register. The file format is correct. Something between "registered" and "launched" is missing — possibly a feature flag, an enable toggle we didn't find, or a timing issue. Don't accept "it can't work" without re-investigating.
