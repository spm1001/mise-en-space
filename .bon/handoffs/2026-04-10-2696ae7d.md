## For Claudes to come

The Claude Desktop Cowork plugin upload path has a structural dead zone for stdio MCP servers. The `create-cowork-plugin` skill documents stdio as supported, the `claude plugin validate` CLI accepts both inline and file-reference forms, but the marketplace validator (the one behind the "Upload local plugin" dialog) requires `mcpServers` to be a string file reference like `"./.mcp.json"`, while the runtime (the Claude Code instance the HostLoop launches) only processes inline `mcpServers` objects. Neither format satisfies both validators. The string-ref form uploads cleanly and silently drops the MCP server at runtime — no error logged, just absent from the connected servers list. The inline form fails upload validation with a generic toast. Adding `"type": "stdio"` doesn't help.

The MCPB format (`manifest.json` with `"server": { "type": "uv", "mcp_config": {...} }`) is the only working vehicle for stdio MCP servers in Cowork because the Remote Plugin Manager has a uv-aware lifecycle handler keyed off `server.type` — it runs `uv sync` to materialize `.venv/` before launching. The `.plugin` format has no equivalent. If you're shipping a Python/uv stdio MCP server to Cowork users, ship MCPB. If you're tempted to use `.plugin` instead because it's documented as supported, save yourself the hours and don't.

Two implementation details worth knowing while you're in this territory: any Python plugin bundle needs `.python-version` (or `uv` will pick the system default and trip on transitive deps without wheels — `onnxruntime` from `markitdown[pdf]` is a common one), and `README.md` (hatchling validates `pyproject.toml`'s readme field at editable-install time). All Desktop-spawned processes go through `/Applications/Claude.app/Contents/Helpers/disclaimer` — its PATH includes `/opt/homebrew/bin` but absolute paths to binaries are safer when debugging "command not found" mysteries.
