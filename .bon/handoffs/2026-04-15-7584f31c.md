### Reflection
This was the "writer-first, schema later" session — we deliberately chose to build the form creator without a shared schema module, learning the API's constraints by building against it. The key discovery validated this: the Forms API rejects newlines in titles, which we only found through a live test. A schema-first approach would have designed around the YAML structure without knowing this constraint.

The three-restart cycle (code change → bump → marketplace update → restart) is friction that's specific to MCP development. The code works locally but the plugin cache means changes don't take effect until the full cycle completes. This is documented but still catches you.

Sameer noted that form descriptions render as plain text — markdown bullet characters (`-`, `*`) appear literally, not as formatted lists. He had to manually adjust some formatting in the created forms. This is a Forms API limitation, not a mise bug, but worth knowing.

## For Claudes to come

**The Forms API write path breaks every assumption from the Drive-based creation pattern.** Doc/Sheet/Slides creation is one API call to Drive's `files().create()` with media upload and MIME-type conversion. Forms creation is two calls on a different hostname: `forms.create()` returns an empty form, then `forms.batchUpdate()` adds questions and sets the description. The request schema for batchUpdate uses `createItem` wrappers with `location.index` for ordering — it doesn't mirror the GET response shape, so you can't just round-trip the structure. Item titles cannot contain newlines (400 error), but descriptions can — `_split_title()` handles this by splitting YAML block scalars. The `forms.body` scope is a superset of `forms.body.readonly`, so only one is needed. Form descriptions are plain text, not markdown — formatting characters appear literally.

When debugging Forms API errors (or any Google API error), `raise_for_status()` in the HTTP client swallows the response body. The actual error message (e.g. "Displayed text cannot contain newlines") is in the JSON response body under `error.message`. To see it, bypass the client with a raw `httpx.Client` call or catch the exception and read `response.text` before re-raising.
