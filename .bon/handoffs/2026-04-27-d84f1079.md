## For Claudes to come

**MCP Resources in CC: empirical reference (Apr 2026, mise-en-space).** Mise already ships eight Resources via FastMCP — six static docs (`mise://docs/*`), one live API call (`mise://gmail/labels` → Gmail labels.list rendered as markdown), and one parameterised template (`mise://tools/{tool_name}` for auto-generated tool docs). Tested in this session against a live CC session: `ListMcpResourcesTool` returns the seven concrete URIs but the parameterised template is **invisible to discovery** — you can `ReadMcpResourceTool` it just fine if you construct the URI yourself, but listing won't surface it. `ReadMcpResourceTool` returns content inline in the tool result, which means cold-fetching via Resource dumps the whole payload into context — fine for tiny live state (label lists, quotas) but architecturally wrong for content (a 50-page PDF would torch the window). The right division of labour: **Resources for small live introspection; the existing `fetch` verb (writing to disk) for content acquisition; native filesystem read for steady-state**. The intuition that an "addressable URI scheme" was needed to unify the surfaces was wrong — when we cut Resources for content, the URI scheme lost its primary user too. `{type, id}` carries everything URIs were carrying. Lesson worth generalising: when you add an architectural layer X "because of Y," and Y later gets cut, audit X — it usually goes too. The web URL (`docs.google.com/document/d/abc`) is the human-facing handle if anyone needs one; nothing requires inventing a new scheme.
