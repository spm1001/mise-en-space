## For Claudes to come

**When a method's behaviour doesn't match what its name promises, prefer relocation over documentation.** Today's first cut of `current_user_email()` worked, passed tests, was even subtly clever — lazy cache, opportunistic backfill on first read. The smell I named in hostile review: `to_dict()` is a method whose name promises pure serialisation, but its first call could trigger an HTTP request, force a token refresh, and write to Keychain. The wrong fix would have been to add a comment warning future Claudes ("note: first call may HTTP"). The right fix was to *relocate* the work to a lifecycle event where the shape matched the name — sync client `__init__` is the place where credential loading already happens, where HTTP is expected. Once moved, `to_dict` is unambiguously pure; `current_user_email` is unambiguously a cache read. No comment needed because the structure now tells the story. Comments age out of sync with code; misplaced behaviour ambushes future readers who took the method name at face value. The general test: if you find yourself wanting to write "note: this method may surprise you", consider whether the surprise belongs somewhere else first.

**A subagent's research can be technically accurate but framed for a different audience than yours.** I asked an agent whether Cowork supports marketplace install. The agent answered correctly — yes, and named two paths (Anthropic-curated public marketplace; org admin → GitHub source). I reported these as the answer. Sameer's screenshot revealed a third path: per-user "Add marketplace" dialog accessible to any Cowork user. The agent hadn't lied; it had downplayed the user-level path because it framed the question as "how do organisations distribute plugins?" rather than "how does this individual user install one?". The lesson is sharper than "verify before claiming": the slant was in the *framing*, not the facts. When your user is a single individual and the agent answered as if for an enterprise admin, the same facts imply a different recommendation. Useful sanity check: when summarising research, name the audience the framing assumes ("the agent answered as if you were an org admin; for a single user, the path is..."). Exposing the slant before propagating it stops your recommendation inheriting the agent's misread of the audience.
