- **Eager extraction path lacks the MIME resolver.** An Outlook-sent PDF or PNG tagged `application/octet-stream` won't get inline preview during thread fetch, even though `fetch_attachment` would handle it. The explicit-fetch escape hatch works but the inline-preview promise breaks. Tracked as **mise-dazode**.
- **Walking-backwards under-strips Apple-Mail-style sigs.** The bare-name line above the full-name+title gets preserved as a content boundary. Intentional trade-off (under-stripping is safer than over-stripping content) and documented in the function docstring + commit message. If you see a future bug report along the lines of "mise didn't strip Alice's name," remember: that's the trade. Sigs with URL+phone densities below still strip; only the boundary changes.
- **Aggressive-strip warnings on short replies.** Will fire on legitimately-short replies with long corporate sigs. Watch for warning fatigue.

## For Claudes to come

**Trust briefs that have done diagnostic work, even when they conflict with existing internal tests.** When a field report names the file, the lines, the root cause, and a fix sketch — that brief is grounded in real-world reproduction. Existing tests are often grounded in synthetic shapes from one fixture. If the brief's fix breaks a test, look at the test before looking for an alternative fix: usually the test asserts a *more aggressive* version of correct behaviour, and the field report has just shown that aggression eats real content. The right move is to update the test to the new (less aggressive, more correct) contract, not to find a clever fix that preserves the test's specific shape. The test was a captured snapshot; the field report is the contract.

Concrete example from this session: the zojoma brief said "walk backwards." I saw that walking backwards would break `test_strips_corporate_contact_block` (which asserts the bare-name line above a corporate sig gets stripped) and went looking for an alternative. The alternative I tried — proximity-window — fixed the *thread-with-quoted-history* case in the brief but not the *standalone-msg-with-URL-rich-sig* case the brief also described. Walking backwards + updating the test to "URL+phone block still stripped; bare-name boundary may survive" was the right answer all along. ~15 minutes wasted before I came back to the brief and read it literally.

The shape generalises: synthetic tests encode the test author's mental model of "what good output looks like"; field reports encode "what wrong output happens in real data." When they conflict, the field report wins.

**Operational note for multi-commit sessions in shared/public repos:** Always `git fetch && git log HEAD..origin/main` before starting. Today the MCPB drop (a routine cleanup commit by Sameer) happened upstream while I was working, and my third commit included a `manifest.json` version bump on a file that no longer existed. Caught only when `git push` rejected the fast-forward. The cost of checking upfront is ~5 seconds; the cost of a phantom-file commit propagating is rebase-with-amend gymnastics under pressure.
