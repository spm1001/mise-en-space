## For Claudes to come

**Adding a do() operation drifts more surfaces than the documented 6-step recipe lists.** The recipe (CLAUDE.md "How to Add a New do() Operation") covers impl → dispatch → register → export → resource-docs → tests, and `test_dispatch.py` auto-verifies OPERATIONS/DISPATCH/REQUIRED_PARAMS sync. But the op **name and count** also live in surfaces the recipe doesn't name: `CLAUDE.md` and `README.md` both carry an "N ops: a, b, c…" line, `tools/dispatch.py`'s `DO_DESCRIPTION_FULL`, `docs_do()` in `resources/docs.py`, and the `SKILL.md` operations table. When you add an op, grep the repo for the current count (`grep -rn "N ops\|N operations"`) and the op list, and update every surface — otherwise the cold-start docs say "14 ops" while the tool has 15.

**The vendor boundary is wider than "runtime code" — most root files are vendored too.** `spm1001/batterie` vendors mise's runtime AND `CLAUDE.md`, `README.md`, `CHANGELOG.md`, `skills/`, `pyproject.toml`, `uv.lock`, `instructions.md`, `credentials.json`, `hooks/`, `apps-script/`. The ONLY non-vendored (free-to-commit-unbumped) trees are `tests/`, `docs/` (the subdir), and `.bon/`. So a "docs fix" to `CLAUDE.md`/`README.md` is **ship-coupled**, not free — verify the live list with `ls ~/repos/spm1001/batterie/plugins/mise/` before assuming any root file is safe to commit standalone.

**markdown↔HTML lives in two places for a reason.** HTML→markdown (`markitdown`) is in the optional `extraction` extra — the slim/Cornichon build does without it (falls back to tag-stripping). markdown→HTML (`python-markdown`, `markdown_to_html`) had to go in **core**, because `draft`/`reply_draft` are remote-safe ops that must render in every flavour incl. slim. Direction-of-conversion maps to build-flavour: anything a remote-safe op needs is core; local fast-path extraction is the extra.
