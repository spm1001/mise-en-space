**Before adding an API call, check what the existing fetch already holds and discards.** samono's last_sender/from_me/unread_count cost ZERO extra calls — the per-thread search fields mask already downloaded every message's headers and labelIds; the code only read messages[0]. The expensive-looking enhancement was a read of data already in hand. Corollary the same evening: meduto's invite-state enrichment needs no scope change because calendar.readonly already covers events.list — the constraint you assume (another call, another scope) may not exist.

**Google Calendar's list API hides cancelled events rather than marking them.** `events.list(iCalUID=…)` returns 0 items for a cancelled event unless `showDeleted=true` — absence-of-result is indistinguishable from never-existed (ledger #6's fail-open shape, as an API design). Any event-state code MUST pass showDeleted=true. Full mechanism + probe evidence: docs/2026-07-07-meduto-invite-event-state.md.

**The calendar cap ate the future, and the fix has three legs that only work together.** Google returns events oldest-first; a single capped page therefore fills with last week before reaching tomorrow. Passing the query (q param) shrinks the result set; full-window internal pagination sees everything; nearest-now selection decides who survives a genuine overflow. Any one leg alone re-breaks the "is tomorrow's meeting still on?" case. find_events_for_file still has the disease (mise-kulefi deletes it — it's uncalled).
