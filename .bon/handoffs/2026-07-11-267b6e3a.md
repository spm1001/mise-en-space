**The Google-Doc triage channel is a complete READ round-trip, and its write-back is reply-and-resolve — NOT arbitrary-anchor create.** Google Docs comment anchors (`kix.*` IDs) are minted **only by the Docs editor** when a human highlights text; the Drive API stores any `anchor` string you send **verbatim but never resolves it** (proven empirically: 5 formats — the offset region, Google's own documented line format, and a format-identical `createNamedRange` `kix.*` ID — all read back `quotedFileContent=''`, vs a real human comment's populated value). So Claude **cannot** create a text-anchored comment via the API — and doesn't need to: a **reply** (`comment_reply`, shipped) attaches to the human's existing thread and **inherits the human's anchor for free**, and can `resolve` it. The full loop that works today: human marks up a Doc (ticks + margin comments) → Claude reads ticks (pirozu, 1.6.0) + reads & **locates** comments (jimive, 1.8.0) → does the work → **replies-and-resolves** in-thread.
