## For Claudes to come

**Google Docs checkbox ("tick box") checked-state is NOT in the Docs API — it is export-only.** `documents.get` returns a byte-identical bullet dict for a checked and an unchecked row (`{listId, textStyle:{}}`); a full-JSON, no-field-mask deep scan finds no checked/completed field — only `textStyle.strikethrough`, which is FALSE on checked rows. The `~~` you see on checked rows in the `text/markdown` export is SYNTHESISED by Google's export renderer from the checked bit — it is NOT `textStyle.strikethrough`, so it cannot be read from the API. The only way to read ticks is a rendered export. mise's approach (`adapters/docs.py::_apply_checkbox_states`): detect a checkbox list (a nestingLevel with `glyphType=GLYPH_TYPE_UNSPECIFIED` and no `glyphSymbol`), fetch the `text/markdown` export, `parse_checkbox_markers` in document order, and `annotate_checkbox_states` tags each checkbox paragraph in place (`_mise_checkbox_checked`); the pure extractor reads the tag and renders `- [x]`/`- [ ]`. Positional alignment (Nth checkbox item ↔ Nth export marker) is exact on single-tab docs. A count-mismatch guard renders plain bullets + a warning — never a wrong tick. One extra API call, only on checkbox-bearing docs.

**Drive comments API has a create/list field asymmetry: `mentionedEmailAddresses` is valid for comments.LIST but 400s on comments.CREATE ("Invalid field selection").** Crucially, the comment is STILL created (Google persists the resource, then fails serializing the response fields) — so a create that 400s can look like a failure while actually succeeding. `_SINGLE_COMMENT_FIELDS` in `adapters/drive.py` deliberately omits it; don't "tidy" it back to match `COMMENT_FIELDS`. General lesson: mocked unit tests can't see field-selection validation — a do()-op that hits a real Google write endpoint needs a live smoke against the real API before you trust it.

**Adding a do() op is seam-cheap when its params already exist on do()'s signature.** `do(comment)` needed only `file_id` + `content`, both already declared in server.py's `do()` — so no signature change, and the ledger-#30 drift-guard stayed green automatically. When the params are NEW (e.g. the future anchored-comment `anchor` param, mise-wetera), the #30 dance is mandatory: signature (server.py) + dispatch + description, together — FastMCP generates the schema from the signature, and pydantic silently drops any argument the signature doesn't declare.
