## For Claudes to come

**Mise ships as two IDENTITY flavours (a second flavour axis, distinct from the full/slim BUILD flavours).** `mise` = work (`mit-workspace-mcp-server` / `@itv.com`); `mise-home` = personal/family (`planetmodha-workspace-mcp` / `@planetmodha.com`). The home flavour is produced by `spm1001/batterie/transforms/make-mise-flavour.sh` — a clean, guarded, string-substitution packaging transform (mise runtime unchanged) that rewrites the distinct-identity strings (data-dir, keychain service, `rules/<name>.md`, plugin name/mcpServers key), swaps `credentials.json`, and **fails the build if any ITV-identifying string leaks through**. It's a substitution transform *because* mise's SessionStart hooks carry hardcoded identity strings and run in the session env, so an `mcpServers.env` block can't reach them. **The trap this exists to fix (mise-tatego):** the transform differentiates every *plumbing* surface but leaves every *model-facing* surface (SessionStart warning, `instructions.md` shard, skill name+description) byte-identical — so two same-named "mise" tools coexist in a session and an unauthenticated flavour's (correct) no-token warning reads as "the *other* mise is broken." A competent Cowork Claude filed exactly that misdiagnosis. When a mise "probe bug" report arrives: check whether two flavours are installed (`ls ~/.claude/plugins/data/mise*/token.json`) before believing the probe is wrong. Per-flavour token stores never cross; on Linux there's no Keychain so the file at `data/<name>/token.json` IS the source of truth (the "Keychain storage failed" line on Linux is benign). Full topology in `understanding.md` → "Identity flavours".
