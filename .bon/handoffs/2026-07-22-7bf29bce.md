## For Claudes to come

Three Google-API facts bought with live probes, all load-bearing for mise's collaboration loop. (1) **Google Docs suggested edits**: `documents.get` defaults to inline-mash for editors; `suggestionsViewMode=PREVIEW_SUGGESTIONS_ACCEPTED` returns the suggester's intended text server-side; in inline view every suggestion-tagged textRun carries `suggestedInsertionIds`/`suggestedDeletionIds`, and the insert+delete halves of a replace SHARE one suggestion id (that's how markup mode pairs them); the API never exposes who made a suggestion. (2) **Drive revisions on native Docs**: the Revision resource has no `name` field and `keepForever` is binary-content-only — both accept the write with HTTP 200 and silently ignore it, so a green response proves nothing (verify by read-back); but per-revision `exportLinks` genuinely work (they 307-redirect to googleusercontent with an embedded token — follow redirects), making historical content recoverable even though versions can't be pinned or named. (3) **Gmail drafts**: N draft objects per thread is legal and conversation view renders exactly one inline — and `drafts.list` under a fields mask returns a ZERO-LENGTH body (not `{}`) when there are no drafts, which will crash a naive `orjson.loads`. Process lesson riding on all three: both build briefs this session were wrong or unproven about mechanism while right about the problem — the cheap probe (3–4 API calls against a purpose-built throwaway fixture) falsified one design and validated the other before any code was written, and the fixture doc then served as repro, test fixture, demo, and smoke target for three releases. Build the tiny fixture; probe the mechanism; only then design.
