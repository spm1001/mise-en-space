## For Claudes to come

Google's Drive markdown→Doc import engine cannot produce a code block. Every block form — bare ``` fences, language-tagged, ~~~ fences, 4-space indented — imports as one code-styled run PER whitespace-delimited token (green + Roboto Mono 9pt on words, mono-but-unstyled spaces, a `` PUA marker glyph on each block's first line): per-word pills in the UI, per-word backtick spans on re-export. Two forms import clean, and they compose into the repair: inline code spans become ONE mono run (double-backtick delimiters work too, yielding a single run with literal backticks inside; consecutive internal spaces survive), and backslash hard breaks become in-paragraph `\x0b` line breaks — so a fenced block rewritten as per-line spans + backslash joins imports as a single tight monospace paragraph (`convert_fenced_blocks`, wired into doc create/overwrite). Dead ends probed so nobody re-treads them: `<pre>` strips to plain text; in-span leading indentation migrates out to paragraph indent. Extractor corollary: `\x0b` in textRun content must render as a newline or spans visually glue on fetch — and Google's own markdown export silently drops it, which is why Docs-API structure inspection, not export round-trips, is the right adjudicator for what an import minted. Related boundary, settled at close: slides create has been `not_implemented` since Jan 2026 because Drive has no text→Slides importFormat at all (PPTX/ODP only) — markdown→deck would be a Slides-API construction project, not an import tweak.
